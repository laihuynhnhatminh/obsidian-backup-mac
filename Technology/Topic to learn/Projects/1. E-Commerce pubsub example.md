# Event-Driven E-Commerce Learning Project (FastAPI + Pub/Sub)

A compact checklist for building a **local event-driven e-commerce backend** for learning purposes.  
Goal: understand **microservices, async events, and message brokers** using a simple book store scenario.

---

# 1. Define the System Scope

Design the architecture before coding.

## Services

- Order Service
    
- Payment Service
    
- Email Service
    
- Inventory Service
    

## Event Flow

```
order.created
      ↓
payment.requested
      ↓
payment.completed
      ↓
order.paid
      ↓
email.send_receipt
inventory.reserve
shipment.create
```

Goal: **services communicate through events whenever possible**.

---

# 2. Setup Local Infrastructure

Install required tools.

- Docker
    
- Google Cloud CLI
    

Start the Pub/Sub emulator locally:

```
gcloud beta emulators pubsub start --host-port=localhost:8085
```

Set environment variable so services connect to the emulator:

```
export PUBSUB_EMULATOR_HOST=localhost:8085
```

---

# 3. Create Pub/Sub Topics

Define the event topics used in the system.

Example topics:

```
order.created
payment.completed
order.paid
email.send
inventory.reserve
```

Create topics:

```
gcloud pubsub topics create order.created
```

Create subscriptions:

```
payment-service-sub
order-service-sub
email-service-sub
inventory-service-sub
```

---

# 4. Create Project Structure

Suggested monorepo structure:

```
ecommerce-event-system/

services/
    order_service/
    payment_service/
    email_service/
    inventory_service/

shared/
    pubsub/
    event_schema/

infra/
    docker-compose.yml
```

---

# 5. Build a Shared PubSub Client

Create reusable publisher/subscriber logic.

Example file:

```
shared/pubsub/client.py
```

Responsibilities:

- publish event
    
- subscribe to topic
    
- handle message acknowledgement
    
- handle retries
    

This prevents duplicate logic across services.

---

# 6. Implement Order Service

Endpoint:

```
POST /orders
```

Workflow:

1. Create order in database
    
2. Set status = `PENDING_PAYMENT`
    
3. Publish event
    

Event:

```
order.created
```

Payload example:

```json
{
  "order_id": "123",
  "user_id": "abc",
  "items": [
    { "book_id": "b1", "qty": 1 }
  ],
  "total": 30
}
```

---

# 7. Implement Payment Service

Subscribe to:

```
order.created
```

Workflow:

1. Receive event
    
2. Simulate payment processing
    
3. Publish new event
    

Event:

```
payment.completed
```

Example payload:

```
{
  "order_id": "123",
  "payment_status": "success"
}
```

For simulation:

```
sleep(2)
payment_success = True
```

Workflow:

update order status → PAID

# 12. Add Observability

Add logging for event publishing and consumption.

Example logs:

```
[EVENT RECEIVED] order.created
[EVENT PUBLISHED] payment.completed
```

This helps debug event flows.

---

# 13. Add Idempotency Protection

Event systems may deliver messages **multiple times**.

Consumers should check if an event was already processed:

```
if event_id already processed:
    ignore
```

This prevents duplicate actions such as charging a payment twice.
